# LCARS-HA-Web — UI STANDARDS (v2, governs the custom build)

> Single source of truth for the LCARS look. These specs are renderer-agnostic: the same
> numbers apply whether drawn by the standalone prototype or the tier-3 custom card. Build
> every screen from these so the whole interface stays coherent.
> Updated 2026-07-01 — folded in the LCARS guideline canon (Bracer Jack tutorial) and locked
> the three font sizes.

All sizes are in **LCARS units (u)**; everything scales off one CSS var `--u`.
`U(n)` = `calc(var(--u) * n)`.

---

## 0. LCARS CANON (the rules that keep it coherent)
These come from the classic LCARS design guideline. Treat them as hard rules; deviate only
deliberately and for a stated reason.

1. **The sweep (elbow) is sacred.** Never disfigure it. It is generated by the ported
   `createRadiusPath` algorithm — never eyeballed. (See §6.)
2. **Frame goes thick → thin at every turn, never the same thickness twice in a row.**
   Our frame obeys this: 5u left bar → 0.64u divider via the elbow.
3. **The rounded cap = the period.** A cap terminates a bar, and its "unconnected" nature is
   exactly why a capped shape reads as a button. **Rounded caps ARE the buttons.** (Textbook —
   we are NOT doing square buttons. This can be revisited later, but v1 is by the numbers.)
4. **Flat vector look.** No gradients, emboss, glow, or shadows. Clean fills only.
5. **Consistency over variety.** One font. Three font sizes. Two text colors. Meaningful
   colors only (§3, §4).

## 1. Typography — THREE sizes only (LOCKED)
Exactly three, no others. Consistency is the whole point.

| Token | Value | Used for |
|-------|-------|----------|
| `--fs-title` | **2.4u** | the current activity / main title (e.g. big "LIGHTING"), and large primary data readouts (e.g. `71°F`, `2%`) |
| `--fs-sub`   | **0.55u** | the four main nav buttons, station/system buttons, and panel title bars (sub-headers) |
| `--fs-data`  | **0.40u** | scrolling numbers on the frame (stardate/clock), the small station label, and content detail lines (`SET 72°F`, `%` labels, `FRED CORE`) |

Font: **Antonio** (condensed), uppercase, letter-spacing ~0.06–0.14em.

## 2. Text — TWO colors only
- **Normal:** `--c-text` (soft lilac-white) for all standard text on black/panel.
- **Highlight:** `--c-tangerine` for the active/selected/processing state.
- Text ON a colored button is `black` (buttons are light fills).

## 3. Color — assign MEANING, cap the palette
Every hue means something; no decorative color. Five functional roles + per-system category
color. (Guideline: >5 unrelated colors = the amateur "multi-color LCARS" tell.)

Functional roles:
1. **Active / highlight** — `atomic-tangerine` (peach)
2. **Idle / inactive** — `lilac`
3. **Periphery / info bars** — `chetwode-blue`
4. **Attention / gold** — `canary`
5. **Alert** — `indian-red`

Category coding (each system keeps a consistent hue = its identity): lighting `canary`,
climate `lilac`, video `chetwode-blue`, sound `magenta`, etc. `orange-peel` / `slate-blue`
available as further category hues, used sparingly.

Surfaces: `ebony-clay` = panel background; `black` = screen bg + seams.

## 4. The Invisible Grid — LOCKED (2026-07-03, Patrick)
LCARS conforms to an invisible grid; every UI element aligns to it. Formalized:

- **Master grid unit:** `--u`. **Snap increment: 0.25u** — every element's x, y, width, and
  height is a multiple of 0.25u. No arbitrary pixel values, ever.
- **Two gap constants only:**
  - **Frame seam** `Sp = 0.18u` — black gaps inside frame chrome (divider halves, elbow→segment,
    segment→segment). Uniform everywhere.
  - **Content gap** `Cg = 0.5u` — gaps between panels/clusters/rows in the content area.
    Larger separations = whole multiples of Cg (1.0u, 1.5u), never in-between values.
- **Row rhythm:** interactive rows (buttons, sliders, list rows) sit on a **0.5u vertical
  rhythm**; button height 1.8u + 0.35u gap ≈ one rhythm cell of 2.25u... round to grid: use
  **1.75u button + 0.5u gap = 2.25u cell** OR keep 1.8u buttons with 0.45u gap (2.25u cell).
  Pick per screen but keep the CELL constant within a screen.
- **Exceptions (canon):** text baselines and nav-button *labels* follow type spacing, not the
  grid. Everything else snaps.
- **Enforcement (renderer):** the LaForge renderer exposes only grid-snapped placement
  (`U(n)` with n in 0.25 steps); free-pixel positioning is not part of the API. WHY: makes the
  grid impossible to violate by accident — alignment is correct by construction, like the elbow.

## 6. Buttons & Caps (canon: rounded cap = button)
- Height **1.8u**, font **`--fs-sub` (0.55u)**, text `black`, centered.
- **Standalone button** = rounded cap on BOTH ends (pill).
- **Connected button** (touches a bar/frame) = rounded cap on the FREE end, flat on the
  connected end. A button paired with an activity bar on its left → flat left, rounded right.
- Caps are drawn by the same corner-radius primitive as the elbows (`createRadiusPath`,
  radius = height/2 for a full cap).
- Width uniform per row; abbreviate only on overflow (`ENGINEERING → ENGR`). Full name always
  shown in the identity stack, so terse button labels are fine.

## 7. The Elbow / Frame — LOCKED (2026-07-01, approved by Patrick)
Drawn as a parametric **SVG path** — the true LCARS "swept" shape: thick vertical bar →
concave inner curve → thin horizontal divider → convex outer corner.
NOTE: the border-radius + black-notch method does NOT work (it clamps and produces broken
notches). The SVG path below is the locked approach. See `frame_test.html`.

Params (LCARS units): `Tv` frame width **5**, `Th` divider thickness **0.64**,
`Ro` outer radius **2**, `Ri` inner radius **1**, `W` elbow piece width **6** (divider
segments start here), `H` elbow piece height **2.6**. Coords are multiplied by U (px/unit).

Bottom-left elbow (upper frame, e.g. lilac):
`M0 0 L{Tv} 0 L{Tv} {H-Th-Ri} Q{Tv} {H-Th} {Tv+Ri} {H-Th} L{W} {H-Th} L{W} {H} L{Ro} {H} Q0 {H} 0 {H-Ro} Z`

Top-left elbow (lower frame, e.g. tangerine) = vertical mirror:
`M0 {H} L{Tv} {H} L{Tv} {Th+Ri} Q{Tv} {Th} {Tv+Ri} {Th} L{W} {Th} L{W} 0 L{Ro} 0 Q0 0 0 {Ro} Z`

The two elbows stack at a seam; their vertical bars extend to meet the full-height frame bars.
Obeys the thick→thin rule (§0.2). **Black seam spacing** `Sp` = **0.18u**, used uniformly:
between the two divider halves, between the elbow and the first segment, and between segments.
Elbow shape + spacing both LOCKED.

## 8. Animation discipline (why we went custom)
- **No obvious loops.** Never let animations restart in unison.
- Give each animated element its **own timeline with a different duration**; stagger start
  delays. (Our breathing already staggers — extend this to everything.)
- Prefer **code-driven** motion/randomness over fixed keyframes where it reduces repetition
  (scrolling numbers, highlights, scans).
- Breathing pulse spec: opacity 1 → ~0.18 (bars) / ~0.5 (buttons, so labels stay legible),
  3–4s ease-in-out, staggered per element.

## 9. Build note
The tier-3 custom renderer implements all of the above with its own SVG element primitive
(the ported `createRadiusPath`). The old ha-lcars `btn()`/`bar()` helpers are superseded but
the SPECS here are unchanged — same units, same radii, same colors.
